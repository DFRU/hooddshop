### P0 — blocks launch / hard-rule violation

#### F-01 — `MadeInUSA.tsx` exists and contains the literal "MADE IN THE USA" string
- **Observation:** `components/ui/MadeInUSA.tsx` exists. It hard-codes the string `"MADE IN THE USA"` twice (lines 33 inline variant, 50 badge variant) and ships an SVG US-flag icon. It is currently unreferenced anywhere in the app (grep for `MadeInUSA` returns only the file itself). The supporting infra — `lib/geo.ts` (`isUSVisitor()`, `getCountryFromCookie()`) and `middleware.ts` (sets `geo_country` cookie via Vercel/Cloudflare headers) — exists *only* to drive this dead component.
- **Impact:** Direct violation of the prompt's hard rule "Never reintroduce 'Made in USA' language". Even though the component is not rendered, the string sits in the source tree and will appear in any source-map served by Vercel if the file is ever imported. PrintKK fulfills overseas — the claim is factually false and creates legal risk (FTC "Made in USA" standard, 16 CFR Part 323).
- **Recommendation:**
  1. Delete `components/ui/MadeInUSA.tsx`.
  2. Delete `lib/geo.ts` (no other consumers).
  3. Delete `middleware.ts` (only used to set the cookie this component reads). Removing middleware also eliminates one Edge invocation per page request — small CWV win.
  4. Remove any unused `next/server` matcher state.
  5. Re-run `grep -rni "made.in.usa\|made in america" .` after deletion to confirm zero hits in `app/`, `components/`, `lib/`, `public/`.
- **Effort:** S

#### F-02 — Analytics IDs are placeholders; no telemetry will be captured at launch
- **Observation:** `components/layout/Analytics.tsx` reads `NEXT_PUBLIC_GA_ID` and `NEXT_PUBLIC_META_PIXEL_ID` and refuses to inject scripts when they equal the placeholder values `"G-XXXXXXXXXX"` / `"000000000000000"`. `.env.local` does not appear to set them (variable list at audit time: `SHOPIFY_STORE_DOMAIN`, `SHOPIFY_STOREFRONT_ACCESS_TOKEN`, `NEXT_PUBLIC_SHOPIFY_STORE_DOMAIN`, `NEXT_PUBLIC_SHOPIFY_STOREFRONT_ACCESS_TOKEN`, `SUPPLIER_ADMIN_TOKEN`).
- **Impact:** If launch happens before tournament traffic, day-one purchases, ad-attribution, and conversion-rate baselines will be unmeasured. Re-attribution after the fact is not possible.
- **Recommendation:** Decide on analytics stack (GA4 + Meta Pixel as currently scaffolded, or alternative such as Plausible / Shopify-side tracking only). If keeping current scaffolding, populate both env vars in Vercel's project settings and `.env.local`. Confirm cookie-consent posture (current site has none) — required in EU/UK before scripts fire if you ship to those regions.
- **Effort:** S (configuration only); M if a consent banner is added.

### P1 — ship before tournament

#### F-04 — PDP price math is inconsistent between desktop CTA and mobile sticky CTA
- **Observation:** In `ProductDetailClient.tsx`:
  - Desktop price (line 198): `${selectedFulfillment ? selectedFulfillment.price_usd.toFixed(2) : displayPrice.toFixed(2)}` — when a fulfillment option is selected, the displayed price becomes the option's `price_usd`, with the original `displayPrice` shown line-through.
  - Mobile sticky CTA (line 294): `${displayPrice.toFixed(2)}` — never updates with the fulfillment selection.
- **Impact:** Mobile shoppers see one price; tapping "Add to Cart" charges the fulfillment-adjusted price. Trust hit and likely refund / chargeback driver. Also: `selectedFulfillment.price_usd` *replacing* the product price (rather than being added on top) is itself ambiguous — is `price_usd` the bundled total, or just shipping? The fallback API response (`app/api/suppliers/options/route.ts` lines 38–53) sets `price_usd: 44.99` while the product price is $49.99, suggesting the option price is intended as the *total* including shipping. This needs a written contract.
- **Recommendation:** (a) Decide whether `FulfillmentOption.price_usd` is total-including-shipping or shipping-only. Document in `lib/suppliers/types.ts`. (b) Apply the same logic to the mobile sticky CTA — single source of truth. (c) Whichever convention is chosen, label the displayed number on the PDP to make it unambiguous ("Total — ships from X" vs "Shipping").
- **Effort:** S

#### F-05 — PDP gallery aspect ratio mismatch causes center-cropping of mockup images
- **Observation:** `ProductDetailClient.tsx` line 137 sets `aspectRatio: "4/3"` for the main gallery image. Mockup files in `public/vehicles/{code}_mockup_*.webp` are 1200×1200 (1:1). With `object-cover`, top and bottom of every mockup are cropped on desktop.
- **Impact:** Visible UX bug — important parts of the printed design (often the upper hood band) get cut off in the hero PDP image.
- **Recommendation:** Change container to `aspectRatio: "1/1"` to match mockup native ratio, *or* switch to `object-contain` with a dark padded background, *or* serve a 4:3 crop variant for the gallery. Recommend the first option (1:1) to preserve detail.
- **Effort:** S

#### F-06 — `/api/products` `reverse` parameter is silently ignored
- **Observation:** `app/api/products/route.ts` line 17: `const shopifySortKey = reverse ? sortKey : sortKey;` — both branches are identical. The `reverse` flag is read from the URL but never reaches `getProducts`. `lib/shopify.ts` does not accept a `reverse` arg. `ShopClient.tsx` lines 99–102 sends `reverse=true` for `PRICE_DESC`.
- **Impact:** "Price: High → Low" sort silently returns the same order as "Price: Low → High". Customers cannot sort descending.
- **Recommendation:** Add `reverse?: boolean` to `getProducts` opts, pipe through to GraphQL. The `products()` connection in Storefront API supports `reverse: Boolean = false`. Update `GET_PRODUCTS` query to accept `$reverse: Boolean` and pass it.
- **Effort:** S

#### F-07 — Footer "Nations" region links all point to plain `/nations`
- **Observation:** `Footer.tsx` lines 46–50 render `["Americas", "Europe", "Africa", "Asia-Pacific", "Middle East"]` as links, all with `href="/nations"`. Region filter state lives in `NationsClient` and is not URL-driven for region (only `?design=` is consumed).
- **Impact:** Broken UX promise — clicking "Africa" should show African nations. Currently it dumps users on the unfiltered grid.
- **Recommendation:** Either (a) make `region` URL-driven in `NationsClient` (`?region=Africa`) and update Footer to use `/nations?region=...`, or (b) remove the region links from the Footer until they work.
- **Effort:** S

#### F-08 — No privacy policy, terms, returns, or FAQ pages exist
- **Observation:** Only `/about` exists alongside the commerce routes. `Footer.tsx` "Info" column has only `Contact` (mailto). Sitemap lists only `/`, `/shop`, `/nations`, `/about` plus dynamic routes.
- **Impact:** (a) Shopify's standard checkout will surface "no policies set" in test mode and reduces customer confidence. (b) Required for Meta/Google ads policy compliance — running paid traffic without a posted privacy policy is grounds for ad-account suspension. (c) Returns policy is required for many credit-card chargeback dispute defenses. (d) GDPR / CCPA / Quebec Law 25 obligations attach the moment the site collects an IP via Vercel/CDN logs.
- **Recommendation:** Stand up `/privacy`, `/terms`, `/returns`, `/shipping`, `/faq` as static MDX or Markdown-driven pages. Link from Footer. Coordinate copy with whoever signs off on legal language — do not write final policy without owner sign-off.
- **Effort:** M (page scaffolding S; legally-reviewed copy is the long pole).

#### F-09 — CartDrawer has multiple accessibility blockers
- **Observation:** `components/layout/CartDrawer.tsx`:
  - No `role="dialog"` / `aria-modal="true"` / `aria-labelledby` on the drawer container.
  - Backdrop click closes the drawer; **no Escape-key handler**.
  - **No focus trap** — keyboard users can tab into the page beneath while drawer is "open".
  - **No focus restore** — when closed, focus does not return to the cart trigger button in `Nav`.
  - Backdrop and drawer share the same `bg-black/60` overlay; no contrast issue, but no `aria-hidden="true"` on background content.
- **Impact:** WCAG 2.1 AA failures (2.1.1 Keyboard, 2.4.3 Focus Order, 4.1.2 Name/Role/Value). Screen-reader users will hear background content while the modal is open. Keyboard-only users cannot dismiss without clicking.
- **Recommendation:** Use a headless-dialog primitive (Radix UI Dialog or Headless UI Dialog) or implement the four standard dialog behaviors manually: ARIA attrs, focus trap, Escape handler, focus restore. Do not introduce a new dependency without explicit approval per the prompt's hard rules.
- **Effort:** S manually, M if introducing a primitive.

#### F-10 — PDP renders Shopify HTML with `dangerouslySetInnerHTML` without sanitization
- **Observation:** `ProductDetailClient.tsx` line 212 inserts `product.descriptionHtml` directly. Shopify product descriptions are HTML and can be edited by anyone with admin access; third-party Shopify apps can also write to them.
- **Impact:** XSS surface. Risk depends on how tightly admin access is controlled, but the safer posture is "treat all CMS-authored HTML as untrusted."
- **Recommendation:** Pipe through DOMPurify on the server (call from the PDP server component, not the client) or use `react-dompurify`. Alternatively, switch to a Markdown-based description model and render with a known-safe parser. Sanitization is a one-line addition; would be a new dependency — needs approval.
- **Effort:** S

#### F-11 — `next.config.ts` ships no security headers
- **Observation:** Only the `/vehicles/:path*` cache header is set. No `X-Content-Type-Options`, `Referrer-Policy`, `Permissions-Policy`, `Strict-Transport-Security` (Vercel sets HSTS for *.vercel.app but not custom domains by default), `X-Frame-Options` / `frame-ancestors`, or CSP.
- **Impact:** Default-permissive headers; clickjacking, MIME-sniffing, and cross-origin information-leak vectors are open. Lighthouse "Best Practices" will dock points. Some payment processors require HSTS on the checkout origin.
- **Recommendation:** Add a global `headers()` rule for `/(.*)`. Minimum starter:
  - `Strict-Transport-Security: max-age=63072000; includeSubDomains; preload`
  - `X-Content-Type-Options: nosniff`
  - `Referrer-Policy: strict-origin-when-cross-origin`
  - `X-Frame-Options: DENY` (or `frame-ancestors 'none'` via CSP)
  - `Permissions-Policy: camera=(), microphone=(), geolocation=(), interest-cohort=()`

  CSP is a larger project (analytics inline scripts, Shopify checkout iframe, fonts.googleapis.com) — recommend deferring to a P2 task with proper testing. List provided here as starter, not as something to implement without reading the policy implications first.
- **Effort:** S for the basic five; M for CSP.

#### F-12 — Shop region/Middle East mapping is fragile
- **Observation:** `ShopClient.tsx` lines 119–148: `REGION_TAG_MAP` and `MIDDLE_EAST_NATIONS = ["saudi arabia", "iran", "iraq", "jordan", "qatar"]` are hardcoded in the component. The Asia / Asia-Pacific / Middle East distinction depends on a substring match against the lower-cased product tag list.
- **Impact:** Adding a new ME nation (Algeria? Tunisia? Morocco listed under Africa already) silently buckets it wrong. The same logic is duplicated nowhere else but the source of truth (`lib/nations.ts`) already encodes `region` per nation.
- **Recommendation:** Drive region filtering off the canonical `NATIONS` array (already imported). Map Shopify product → nation (via `getNationCodeFromTitle`), then filter by `nation.region`. Single source of truth, eliminates the brittle tag substring matching.
- **Effort:** S

#### F-13 — Documentation files explicitly required by First Actions are partly absent or thin
- **Observation:** `CLAUDE.md` is a 51-line summary. `AGENTS.md` is 6 lines saying "this is not the Next.js you know — read `node_modules/next/dist/docs/`." `HANDOFF.md` (308 lines) is the only meaningful long-form context, and it is stale (see Section 1). No `CHANGELOG.md`, no architecture doc, no env-setup guide other than the section in `HANDOFF.md`.
- **Impact:** Onboarding a new contributor (or restarting this session on a fresh machine, as the prompt anticipates) will require re-deriving facts from the code. Risk of repeated mistakes.
- **Recommendation:** Promote the audit-time state notes (Section 1 of this file) into `HANDOFF.md`. Add a one-page `ARCHITECTURE.md` covering: route map, data flow (Shopify → server component → client), supplier engine, image pipeline. Defer until P0/P1 fixes ship.
- **Effort:** M

### P2 — post-launch / next pass

#### F-14 — Raw `<img>` tags bypass `next/image` optimization in 4 locations
- **Observation:** `Hero.tsx` line 97 (flag), `VehicleShowcase.tsx` lines 59 + 101 (flags), `CartDrawer.tsx` line 85 (Shopify cart-line image), `nations/[code]/page.tsx` lines 90 + 96 (product photo + flag), `NationsClient.tsx` line 111 (flag).
- **Impact:** Larger payloads, no AVIF/WebP negotiation, no automatic responsive `srcset`. `flagcdn.com` and `cdn.shopify.com` are already in `next.config.ts` `remotePatterns` — the configuration is ready, the tags just don't use it.
- **Recommendation:** Convert each to `<Image fill sizes="..."/>` with explicit width/height where the layout is fixed. For `nations/[code]/page.tsx` the local product photo also needs explicit dimensions — check `lib/vehicles.ts` for the source values.
- **Effort:** S

#### F-15 — Per-route `loading.tsx` files are absent
- **Observation:** Only the homepage uses inline `<Suspense>` boundaries. `app/products/[handle]/`, `app/shop/`, `app/nations/`, `app/nations/[code]/` have no `loading.tsx`.
- **Impact:** During Shopify fetch (`revalidate: 60` server, no cache client), the page blocks until response. On slow networks the user sees a blank screen between navigation start and first paint.
- **Recommendation:** Add `loading.tsx` per route with skeletons matching the route's hero region. Reuse the skeleton patterns already in `app/page.tsx`.
- **Effort:** M

#### F-16 — `ProductJsonLd` is minimal — likely no rich-snippet eligibility
- **Observation:** Schema includes `name`, `description`, `image`, `brand`, `sku`, `offers`. Missing: `aggregateRating`, `review`, `priceValidUntil`, `itemCondition`, `mpn` or `gtin`.
- **Impact:** Google Search Console will flag missing recommended fields. Product rich snippet (price, availability, rating stars in SERP) may not render until at least `priceValidUntil` and condition are present, and rating-stars require a real review source.
- **Recommendation:** Add `priceValidUntil` (e.g., end-of-quarter or end-of-tournament window), `itemCondition: "https://schema.org/NewCondition"`. `aggregateRating` requires real reviews — defer until a review system is live. Do not fabricate ratings.
- **Effort:** S

#### F-17 — `sitemap.ts` has no images sub-sitemap
- **Observation:** Standard sitemap only. Next.js 16 supports image sitemaps via the `images` field on each entry.
- **Impact:** Google Image Search will crawl images via OpenGraph and inline references but won't have explicit nation→image hints.
- **Recommendation:** Add `images: [primaryProductImageUrl]` per product entry. Low effort, modest SEO upside.
- **Effort:** S

#### F-18 — `FulfillmentSelector` country picker covers ~14 countries
- **Observation:** `COUNTRY_FLAG` / `COUNTRY_NAME` lists 14 countries. Visitors from anywhere outside this list see a globe emoji and cannot manually switch (their country is not in the dropdown).
- **Impact:** UX for international long-tail (Greece, Argentina, Saudi Arabia, etc. — many of whom are exactly the target market) is degraded. Ironic given the World-Cup theme.
- **Recommendation:** Hydrate the country list from a complete ISO-3166 dataset (or at minimum the 48 `NATIONS` codes). The fulfillment router will return whatever options are available regardless. The flag emoji can be derived from the country code algorithmically (`String.fromCodePoint(0x1F1E6 + (c.charCodeAt(0) - 65), 0x1F1E6 + (c.charCodeAt(1) - 65))`).
- **Effort:** S

#### F-19 — `NationsPage` fetches all 250 products on every request only to map handles
- **Observation:** `app/nations/page.tsx` calls `getProducts({ first: 250 })` server-side, then iterates 48 nations × ~48 products to build `nationProductMap`. Server-side ISR cache (60s) softens this, but the wire payload is large (48 products × 3 image edges × ~10 fields each).
- **Impact:** Slower TTFB on cold builds. Larger Vercel bandwidth.
- **Recommendation:** Either (a) pre-build the nation→product map at build time as static JSON and import, or (b) use a smaller GraphQL query (just `handle`, `images.edges.first 1`) for this specific page.
- **Effort:** M

#### F-20 — `lib/queries/collections.ts` may be unused
- **Observation:** File exists but I did not confirm its consumers in this audit pass.
- **Impact:** Potential dead code. Verify.
- **Recommendation:** `grep -rn "from .@/lib/queries/collections."` and delete if unreferenced.
- **Effort:** S

#### F-21 — `console.warn` in `lib/shopify.ts` will spam Vercel logs on token misconfiguration
- **Observation:** Three `console.warn` calls fire on missing token, non-200 response, and fetch error.
- **Impact:** In production these become unstructured Vercel function logs, hard to alert on.
- **Recommendation:** Replace with structured logging (key/value pairs at minimum) and silence the "no token" warning to a single startup log rather than per-request. Defer until log volume is actually a problem.
- **Effort:** S

#### F-22 — Cart drawer has no slide-out animation; cart line price assumes USD
- **Observation:** `CartDrawer.tsx`: only `slide-up` keyframes. `parseFloat(line.merchandise.price.amount)` (line 68) and the `$` symbol throughout — no currency-symbol mapping.
- **Impact:** Mild UX polish miss; if Shopify is configured for multi-currency (`@inContext` is in the queries), prices will be parsed correctly but rendered with the wrong symbol.
- **Recommendation:** Add `slide-down` exit animation (defer mount unmount with state). Use `Intl.NumberFormat(navigator.language, { style: 'currency', currency: line.merchandise.price.currencyCode })`.
- **Effort:** S
